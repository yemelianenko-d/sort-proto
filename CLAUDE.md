# CLAUDE.md — sort-proto

«Сортуй!» — technical puzzle prototype: мобільна sorting-головоломка на
**Phaser 3 + TypeScript + Vite** (обгортка Capacitor). Мова спілкування з
розробником — українська. Коментарі в коді та commit messages — англійською.

## Обов'язково прочитати перед роботою

- `ARCHITECTURE.md` — шари, контракт модуля механіки, lifecycle рівня.
  Напрямок залежностей порушувати не можна: механіка спілкується назовні
  **тільки через EventBus**, ніколи не імпортує platform/browser API.
- `docs/STYLE_GUIDE.md` — візуальний style-гайд (стрижень «зошит у
  клітинку», палітра, sketch-рендер, матеріальні метафори механік,
  UI-кит, рецепт «додати візуал нової механіки в стилі»). Читати перед
  будь-якою візуальною/UX-роботою. Пов'язані: `docs/asset-spec.md`,
  `docs/mockup-spec.md`.
- `graphify-out/GRAPH_REPORT.md` (якщо існує) — карта кодової бази перед
  архітектурними питаннями. Після змін коду: `graphify update .` (без API-витрат).

## Команди

```bash
npm run dev              # dev-сервер (Vite)
npm test                 # vitest (наразі 138 тестів — усі мають бути зелені)
npm run lint             # eslint src tools
npm run build            # tsc --noEmit && vite build
npx vite build --config vite.standalone.config.ts && node tools/make-standalone.mjs
                         # standalone-білд одним HTML-файлом (для швидкого шарингу)
npm run levels:generate  # регенерація рівнів
npx tsx tools/analyze-levels.ts   # звіт розподілу механік по декадах
```

## Quality gates (перед кожним пушем)

1. `npx tsc --noEmit` — чисто
2. `npm test` — усі тести зелені
3. `npm run lint` — чисто
4. `npm run build` — успішний
5. Після змін у генераторі рівнів — прогнати аналізатор і перевірити
   розподіл механік по декадах

## Стан проекту (липень 2026)

**Зроблено (останнє — v8 «зона напруги» + фінальні анімації, липень 2026):**
- **Баланс v8 — модель «зони напруги»** (відповідь на «чому BB цікавий, а ми
  ні», доведено `diagnose.ts`): (1) кожен рівень стартує на авторському
  симетричному патерні 25–44% (бібліотека `PATTERNS`: cradle/checker2/comb/
  steps/frame/walls/diamond/gate) — перший хід уже пазл, дошка живе на
  ~45% заповнення; (2) генерація на користь `SOLVABLE_AFTER_CLEAR`
  (0.38→0.68 по бандах) — кожен пул = міні-головоломка «зачисти, щоб
  влізло»; (3) `calibrate.ts` переведено на CASUAL-автоплеєра (клік по
  зачистці, неідеальні місця) — цілі чесні для людини: score-драбина
  80→150, collect-квоти 2–4, рівень = спринт 8–15 фігур. Casual-він-рейти
  виміряні: TUTORIAL 88%, EASY ~50-85%, HARD ~37-63%. Справжні марафони
  (квоти 8+) чекають progress-persist.
- **Фінал рівня як у BB**: `playWinCelebration` у сцені — тайли дошки
  розлітаються хвилею від центру у своїх кольорах + конфеті-дощ ~54
  квадратики палітри по всьому екрану; попап приходить після феєрверку
  (1.25с). Панель очок: арт-рамка збережена, запечені трек/риски/квадрати
  вирізані тулзою — шкала малюється грою рівно між колами (не налазить),
  11 рівних рисок, «Очки» 26px. Спец-тайли: замальовка старого символу
  розширена (inset 13% + темні пікселі) — прибрані «дивні риски». Зелена
  галочка `icon_check` (ассет розробника, shared) замість «0» у чипі.

**Зроблено (баланс v7 з симуляції + варіативність фігур, липень 2026):**
- **Каталог фігур v4**: додано 4 S/Z-тетроміно (`SZ_S_H/Z_H/S_V/Z_V`, family
  `SKEW`); L-тетроміно (`L2x3/L3x2`) DEMANDING→NORMAL (стандартні, часті в BB).
  43 форми. S1 в каталозі, але БЕЗ використання в кампанії.
- **`master`-ростер** — один широкий пул (усі тетроміно L/J,S/Z,T,I,O + кути +
  прямокутники, БЕЗ S1) як бекбон майже всіх рівнів. ~60% розіграшів тепер
  кутові/T/S-Z/L (було ~0). Варіативність = виживання. themed-ростери лише для
  явних «Big Blocks».
- **Керована генерація, НЕ honest** (ключове з симуляції): чесна на 8×8 занадто
  жорстока (autoplayer мре за ~35 ходів). NORMAL+ повернуто на КЕРОВАНУ (висока
  `SOLVABLE_NOW`); складність від розміру цілі + tierMix + малої DANGEROUS.
- **Цілі відкалібровано `tools/blocks/calibrate.ts`** (survival-autoplayer per
  band). Реальна стеля score ≈350 → драбина 130→~330 (НЕ 1650). Score-рівні на
  master (bigscore шкодить — великі фігури заповнюють→смерть). Марафони 4–7.
  `tools/blocks/sim-survival.ts` — димова перевірка по всіх 40.
- **`openingBatches` тепер читається генератором** (авторський перший лоток).
- **Візуал**: нові контурні `symbol_N` + `special_N` (дерев'яна база + БІЛЬШИЙ
  символ, `?v=3`). Панель очок ПОВНІСТЮ процедурна (рамка+трек+11 рисок+2 кола,
  «Очки» нижче) — арт-панель не використовується. Спец-тайли — універсальна
  янтарна підсвітка (`SPECIAL_HIGHLIGHT`). Веселку мульти-зачистки ПРИБРАНО —
  кожен тайл лопається у своєму кольорі (як BB).

**Зроблено (кампанія v6 + гайдлайн левел-дизайну, липень 2026):**
- **`docs/BLOCKS_LEVEL_DESIGN.md` — джерело правди балансу блоків** (аналіз
  усіх референсів). Ключовий принцип: «порожнеча = безпека, великі фігури =
  темп» — ранні дошки ~95% порожні (лише авторські симетричні пресети, БЕЗ
  runway-рядків і random-сміття), складність ранніх рівнів = розмір квоти,
  НЕ тісна геометрія (смерть на туторіалі = помилка дизайну; розробник
  програв v5-L1 чотири рази — виправлено). Правки балансу: спершу гайд,
  потім `build-campaign.mjs`.
- **`openingBatches` реалізовано в генераторі** (валідувалось парсером, але
  ІГНОРУВАЛОСЬ раніше — діра): `generateBatch(..., batchIndex)` віддає
  авторські перші батчі. L1–L3/L6 мають прописані перші лотки — перший хід
  ніколи не буває незручним.
- **Кампанія v6**: L1–2 score на ПОРОЖНІХ дошках з opening-батчами; L3–5 —
  каскад {0:4}→{0:4,1:4}→{0:5,1:5,2:6} із пресетами, скопійованими з
  референс-скрінів (4–6 тайлів, симетрія); всі runway прибрано зі score-
  рівнів (порожні + scatter на NORMAL+); ростер `starter` покрупнішав
  (R3x3 у ньому, середня фігура ≥4 кл.). Комбо-напис — «Комбо N» (ціле
  число ланцюга, без множника — референс-формат).

**Зроблено (кампанія v5: щільність подій + каскад символів, липень 2026):**
- **Фікс «нудних ранніх рівнів» (v5).** Діагноз: нудьга = низька частота
  значущих подій, не розмір чисел. Рішення в `build-campaign.mjs`:
  (1) жирні ростери скрізь — 3×3 повноцінний з L1 (tierMix TUTORIAL/EASY
  відкрито: demanding 0.13–0.17), новий hero-ростер `bigstart` (R3x3 лідер)
  з per-level `tierMix`-override `BIG_MIX` (demanding 0.32) — рівні
  «Big Blocks» L6/L14 як у референс-скріні з трьома 3×3 у лотку;
  (2) потік спешлів на майже кожному батчі: `targetBatchChance` 0.7–0.9
  ВІД L3, дефолти mb2/mp2 (мульти-спешл фігури);
  (3) score-дошки = «runway» (хелпер `runway(h,v)` — рядки/стовпці
  заповнені на 4–6/8, зачистки і ланцюг з 1–2 ходу);
  (4) **каскад символів як у референсі**: L1–2 score-онбординг →
  L3 {0} → L4 {0,1} → L5 {0,1,2} → L7/L8 інтро символів 3 і 4 —
  це і є шаблон початку петлі (спостереження розробника: у референсі
  «синій → синій+помаранчевий → +жовтий», старт із двох score-рівнів).
  Бейкер валідує інваріанти (повні лінії, supply≥quota) при кожному бейку.
- **Знахідки з відео-аналізу референсу:** (а) фейл марафону → реклама →
  прогрес квот ЗБЕРІГАЄТЬСЯ, скидається лише дошка (війна на виснаження,
  не один забіг); (б) екстремальний urgency-feed: коли лишився 1 символ,
  лоток отримує H5 із 5 однаковими спешлами; (в) після ~L42 цілі
  закільцьовуються зі зростанням чисел (template loop). Кандидати на
  впровадження: progress-persist на марафонах, фінальний «злив» символу,
  генерація петлі L41+.

**Зроблено (кампанія v4: 40 рівнів за референсом Block Blast, липень 2026):**
- **Комбо = довжина ланцюга** (`DEFAULT_SCORE_POLICY`: `comboStep 1.0`,
  `comboMax 8`): ланцюг 3 зачисток → ×3. Відкалібровано за відео референсу
  (база трикутна 10/30/60 у них ТА САМА). Головна стратегія score-режиму —
  тримати ланцюг: 6 одиночних у ланцюзі = 210 очок проти 60 без.
- **Мульти-спешл фігури** (несуча конструкція марафонів): `TrayPiece.special`
  → `specials[]`, економіка роздає до `maxTargetsPerPiece` на фігуру та
  `maxTargetsPerBatch` на батч (циклом за urgency, без колізій клітинок,
  без пере-постачання квоти). З відео: марафон 5×32 цілі за ~50 батчів
  вимагає ~3 спешли/батч — з лімітом 1/фігуру математично неможливо.
- **Score-прогрес-бар** замість тексту (`buildScoreBar` у сцені): бульбашка
  з рахунком їде по треку до цільового кружка, пульсує на кожному прирості.
  Референсний felt-pace: та сама цифра на барі відчувається швидшою.
- **Кампанія 40 рівнів** (`build-campaign.mjs`, v4): LEARN 1–6 (score-forward,
  L1=130 — калібрувальна точка розробника з референсу) → PRACTICE 7–14
  (чесність з L7) → EXPAND 15–22 → DEEPEN 23–30 → **SCORE-КЛАСТЕР 31–33**
  (три поспіль, як у референсі) → **МАРАФОНИ 34–40** (2–5 типів × 6–8,
  фінал L40 = всі 5 символів, chance 0.9, mb3/mp2). Score-драбина:
  130→170→220→320→420→520→650→800→950→1100→1250→1400→1650. Мікс: 27 collect
  / 13 score. Пресети — авторські симетричні патерни. Інваріант
  supply≥quota перевіряється при бейку. Похвальні написи (`blocksPraise.ts`):
  2/3/4+ ліній + «Чистий аркуш» (all-clear), стакаються з комбо-бейджем.
- **Кампанія стала челенджовою (v3).** З NORMAL-банду (L7+, крім EASY-передишок)
  генерація ЧЕСНА — `batchPolicy.honest: true` у `BANDS` білдера: без
  solvability-гарантії, дошка може реально вбити рівень → програш → ретрай
  (детермінований сід per attempt лишився — ретрай чесний). TUTORIAL/EASY
  гарантовано-розв'язні (онбординг). Цілі ВКОРОЧЕНІ (рівень = спринт ~12–20
  фігур, не гринд): квоти зрізані (5-6 → 3-5), `targetBatchChance` піднято
  (0.34 → 0.45–0.55), score-таргети зменшені (~20-25%), пресетів інколи
  більше за квоту (вибір, які збирати). Дошки NORMAL+ стартують ЩІЛЬНІШИМИ:
  новий хелпер `scatterBlocks(board, seed, count)` у `build-campaign.mjs`
  (детермінований mulberry32, кап ≤5 заповнених на рядок/стовпець — лінія
  ніколи не стартує повною), count 6→14 із складністю. Парсер валідує
  `batchPolicy.honest` (boolean). У Block Blast НЕМАЄ ліміту фігур — челендж
  саме від чесного рандому + заповнення поля, не від лімітів ходів.
- **Endless-режим ЗАКОНСЕРВОВАНО** (рішення розробника: «зараз не потрібен,
  тільки лінійні рівні»). Код лишився робочим і повністю ізольованим
  (`blocksEndless.ts`, `?mechanic=blocks&endless=1`), але ніде не
  експонується — без плитки/посилань. Не видаляти і не розвивати без запиту.

**Зроблено (Endless-режим + ребаланс опенінгу «Блоки», липень 2026):**
- **Endless / Arcade режим (справжня петля Block Blast).** Порожнє 8×8, БЕЗ
  цілі, чесна генерація (`BatchPolicy.honest` → `BlocksGenerator` пропускає
  solvability-гарантію, тож дошка може реально зайти в глухий кут = game over).
  Ганяєш high-score + комбо, доки вміщуються фігури. Новий goal `{type:'endless'}`
  (`goalMet`=false завжди), `GoalProgress` endless, вбудований
  `blocksEndless.ts` (`BLOCKS_ENDLESS_CONFIG` — повний ростер, tierMix
  0.42/0.4/0.15/0.03), рекорд у localStorage (`blocks.endless.best`), game-over
  попап (рахунок + рекорд + «Новий рекорд!»). Свіжий рандом-сід щоразу (не
  детермінований, як кампанія). Вхід: **`?mechanic=blocks&endless=1`**
  (флаг `endless` у `readAppFlags`; роут у `Preloader`). Плитка в лоббі —
  коли зʼявиться master-lobby. i18n `mechanics.blocks.endless*` (uk/en).
- **Ребаланс опенінгу кампанії (v2).** Правки в `tools/blocks/build-campaign.mjs`:
  (1) `tierMix` TUTORIAL/EASY відкрито NORMAL-тір (0.3→0.4) — раніше великі
  фігури зануляла tier-вага, хоч би яка вага в ростері; (2) новий ростер
  `starter` (квадрат R2x2 + 3–4-барні/L/T; доміно лише заповнювач) — середній
  розмір фігури L1 2.64→3.23 кл., доміно 40%→10%, 4+клітинок 10%→38%;
  (3) score-forward онбординг: **L1 тепер `score`** (петля «постав→зачисти→очки»),
  collect заходить з L3; мод-мікс 21 collect / 9 score (було 23/7), очки на
  рівнях 1,2,4,7,10,14,19,23,27. Score-таргети — оцінки пейсингу, чекають
  плейтесту. `blocks_levels.json` перебейкано.

**Зроблено (механіка «Блоки» (Block-Blast-стиль), липень 2026):**
- **Механіка `blocks` — другий мешканець Puzzle Hub.** Повний контракт за
  MECHANIC_SDK: `src/mechanics/blocks/` (Types → Parser → Model+тести →
  ViewContract → View → StubView → Controller+тести → Scene → module → index),
  бакет `public/assets/mechanics/blocks/` (8 тайлів 128², ключі `blocks/tile_N`),
  i18n `mechanics.blocks` (uk/en), рядок у `app/mechanics.ts`, CODEOWNERS.
  Геймплей: 8×8 дошка, лоток на 3 випадкові фігури з вагового пулу рівня
  (каталог поліміно `blocksPieces.ts`, без обертань у грі), два способи вводу:
  драг із ghost-прев'ю (підйом над пальцем `dragLiftCells` — ЛИШЕ на тачі,
  мишею фігура під курсором) і тап-вибір → тап-постановка (як у сортингу).
  Повні рядки/стовпці зникають. Ціль рівня — union: `clear` (прибрати
  стартові блоки; на полі НЕ маркуються — рішення розробника, ціль показує
  HUD-чип) або `score`. **Два режими цілей (за дизайном):** `score` (набрати
  N очок, HUD-чип «Очки: S / T») і `collect` (зібрати квоти спец-символів;
  HUD — ряд чипів «символ × лишилось», як у Block Blast). **Спец-тайли**
  (`SpecialSymbol` 0..4: циркуль/кутник/транспортир/лінійка/олівець) —
  дерев'яна база + символ; збираються, коли лінія з ними зникає. З'являються
  двома шляхами: пресети на полі (`level.specials[]`) і **гарантована
  інжекція** в лоток (`topUpSpecials` тримає кожен ще-потрібний символ
  досяжним, щоб collect-рівень не міг зайти в глухий кут). Арт:
  `blocks/special_N` (тайл), `blocks/symbol_N` (значок для HUD), тулза
  `tools/art/prepare-blocks-specials.mjs`. Старий режим `clear` ВИДАЛЕНО.
  Візуал поля — «технічний
  кресленик у зошиті» (затверджений концепт): подвійний контур, розмірні
  лінії з datum-кружками і підписами 8×8, засічки-лінійка, позначки 90°;
  дудли полів — геометрія уроку (циркуль, транспортир, теореми, куб) у
  `blocksDoodles.ts`, процедурні, mechanic-owned (спільні дудли не
  використовуються). «Очки» — над полем, вище розмірної лінії. Фонова сітка
  зошита під полем пригашена paper-wash (alpha 0.82). Лоткові тайли ~85%
  клітинки. Скоринг: клітинки +
  `10·lines²`. Провал = жодна фігура з лотка не вміщується. Рандом
  інжектиться (`mulberry32`) — модель детермінована в тестах. Тюнінг
  механіки — у `blocksSettings.ts` (mechanic-owned, спільний конфіг не чіпали).
  10 рукотворних рівнів у `public/levels/blocks_levels.json`; парсер
  відхиляє передзаповнені лінії; shipped-файл валідується тестом.

## Конвенції

- Модель (`SortingModel`) — чиста логіка без Phaser, покривається юніт-тестами.
- View працює тільки через `SortingViewContract`; для тестів контролера є `StubView`.
- Нова фіча в механіці = зміни в Types → Parser → Model (+ тести) → ViewContract
  → View → StubView → Controller. Не пропускати StubView і контракт.
- Рівні — декларативні (`public/levels`, `sorting_levels.json` генерується тулзами).
- Піксель-точні візуальні правки робити мінімальними кроками (2–3px) і
  показувати білд розробнику після кожної.
- Commit messages: короткі, описові, англійською (див. `git log`).
